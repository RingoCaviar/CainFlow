# 模型多供应商绑定改造计划

## 背景

当前 CainFlow 的模型配置使用单个 `providerId` 绑定一个供应商，节点只保存模型 ID（`apiConfigId`），执行时通过模型反查唯一供应商。

本次目标是改为：

- 一个模型可以绑定多个供应商
- 节点仍然先选择模型
- 当模型绑定多个供应商时，节点显示供应商下拉框供用户选择
- 当模型只绑定一个供应商时，节点自动使用该供应商且不显示下拉框
- 兼容旧配置、旧本地状态、旧工作流

## 总体方案

- 模型配置结构从单个 `providerId` 升级为 `providerIds: string[]`
- 保留旧数据兼容：读取时将旧 `providerId` 自动迁移为 `providerIds`
- 节点序列化新增保存 `providerId`，作为节点级实际执行供应商
- 执行链路优先使用节点保存的 `providerId`，缺失时回退到模型绑定列表中的首个有效供应商

## 阶段拆分

### 阶段一：底层数据结构与兼容迁移

目标：

- 建立多供应商模型结构
- 打通旧数据到新结构的兼容迁移

涉及文件：

- `js/features/execution/provider-request-utils.js`
- `js/core/constants.js`
- `js/core/state.js`
- `js/features/persistence/project-io.js`
- `js/features/ui/ui-controller.js`

任务：

1. 模型归一化逻辑支持 `providerIds`
2. 兼容旧字段 `providerId -> providerIds`
3. 配置导入导出校验改为基于 `providerIds`
4. 默认模型配置切换为多供应商结构

交付物：

- 所有模型配置在运行时统一拥有 `providerIds`
- 旧配置文件、旧本地状态可无报错加载

优先级：

- 最高

依赖关系：

- 无，后续所有阶段依赖本阶段

时间预估：

- 0.5 天

风险：

- 旧导入数据混合存在 `providerId` 和 `providerIds`
- 某些调用点仍然默认读取 `providerId`

缓解措施：

- 统一在归一化函数里兜底
- 全局搜索所有 `providerId` 读取点并逐步替换

验收标准：

- 本地状态加载正常
- 配置导入正常
- 默认模型可正常显示
- 不再因缺少 `providerId` 直接报错

### 阶段二：设置面板模型管理改造

目标：

- 让模型支持绑定多个供应商
- 消除“同模型因供应商不同而重复创建”的旧逻辑

涉及文件：

- `js/features/settings/settings-controller.js`

任务：

1. 将模型卡片中的供应商单选改为复选列表
2. 删除供应商时同步从模型的 `providerIds` 中解绑
3. 拉取供应商模型列表时，优先合并到已有模型而不是重复新建
4. 刷新模型列表和节点下拉时同步使用新结构

交付物：

- 设置面板支持一模型多供应商绑定
- 同一个模型可在多个供应商间复用，不再制造重复模型

优先级：

- 最高

依赖关系：

- 依赖阶段一

时间预估：

- 0.5 天

风险：

- 现有请求示例预览和协议说明依赖单一供应商
- 供应商删除后节点引用可能悬空

缓解措施：

- 预览逻辑增加供应商兜底
- 删除后刷新节点供应商选择并触发回退

验收标准：

- 一个模型可以勾选多个供应商
- 同模型从不同供应商添加时不会重复生成多条模型
- 删除供应商后模型绑定状态正确更新

### 阶段三：节点 UI 与节点序列化改造

目标：

- 节点在模型绑定多个供应商时显示供应商下拉框
- 节点保存所选供应商

涉及文件：

- `js/nodes/node-view-factory.js`
- `js/nodes/node-dom-bindings.js`
- `js/nodes/node-serializer.js`
- `js/features/ui/clipboard-controller.js`

任务：

1. `ImageGenerate` / `TextChat` 节点新增供应商下拉区域
2. 根据模型绑定数量动态显隐供应商下拉
3. 模型切换时同步刷新供应商选项
4. 生图节点协议相关 UI 跟随“节点当前供应商”变化
5. 工作流保存、复制粘贴时保存 `providerId`

交付物：

- 节点可见的供应商选择 UI
- 节点数据中具备模型 ID + 供应商 ID

优先级：

- 最高

依赖关系：

- 依赖阶段一、阶段二

时间预估：

- 0.5 天

风险：

- 现有分辨率/宽高比切换逻辑默认只依赖模型
- 旧节点恢复时未保存 `providerId`

缓解措施：

- 将相关 UI 刷新统一改为基于“模型 + 当前供应商”
- 恢复时回退到首个有效供应商

验收标准：

- 单供应商模型时不显示供应商下拉
- 多供应商模型时显示供应商下拉
- 保存并重新加载后节点供应商选择保持一致

### 阶段四：执行链路与工作流兼容

目标：

- 执行前、执行中、工作流导入时均使用节点级供应商

涉及文件：

- `js/features/execution/execution-core.js`
- `js/features/execution/workflow-runner.js`
- `js/features/persistence/workflow-model-resolver.js`

任务：

1. 执行节点时按节点 `providerId` 解析实际供应商
2. API Key 预检查改为按节点实际供应商检查
3. 旧工作流导入时兼容缺失 `providerId`
4. 供应商失效时给出更准确的缺失提示和回退逻辑

交付物：

- 执行链路不再依赖模型唯一供应商
- 旧工作流仍可继续使用

优先级：

- 高

依赖关系：

- 依赖阶段一、阶段三

时间预估：

- 0.5 天

风险：

- 节点序列化和执行侧选择逻辑不一致
- 工作流导入后节点供应商回退可能让用户困惑

缓解措施：

- 统一供应商解析函数，避免多处重复判断
- 缺失供应商时给出明确提示

验收标准：

- 文生图、对话节点均能按节点选中的供应商执行
- 未选择供应商时能自动回退
- 无有效供应商时提示明确

### 阶段五：联调验证与回归检查

目标：

- 验证多供应商模型的完整交互链路
- 确保未破坏原有配置、工作流和节点行为

涉及范围：

- 前端全链路

任务：

1. 语法检查
2. 配置导入导出回归
3. 工作流保存加载回归
4. `ImageGenerate` / `TextChat` 节点多供应商切换验证
5. 单供应商模型显示逻辑回归

交付物：

- 验证结果
- 已知风险说明

优先级：

- 高

依赖关系：

- 依赖前四个阶段完成

时间预估：

- 0.5 天

风险：

- 某些隐藏调用点仍残留单供应商假设

缓解措施：

- 结合全文搜索和运行前检查补齐遗漏

验收标准：

- `node --check` 前端文件通过
- 关键配置和工作流链路通过手动验证
- 无明显单供应商残留报错

## 关键风险总览

1. 单供应商假设散落在多个模块中，若只改 UI 会导致执行时错用供应商
2. 旧数据兼容是高风险点，必须统一在归一化层兜底
3. 生图节点的协议相关 UI 必须基于“节点当前供应商”，不能只基于模型

## 实施顺序建议

1. 先做阶段一，统一数据结构和兼容迁移
2. 再做阶段二，改设置面板绑定逻辑
3. 接着做阶段三，改节点 UI 和序列化
4. 然后做阶段四，收口执行链路
5. 最后做阶段五，集中回归

## 备注

- 工作流 JSON 仍然不写入全局 `providers` / `models`
- 节点仅额外保存自身所选 `providerId`
- 本次改造以最小破坏兼容为原则，不引入新的全局配置概念
